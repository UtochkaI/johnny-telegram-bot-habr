# Как плохой VPS для VPN случайно превратился в злого Telegram-бота-инженера

Исходные материалы, хронология и sanitized source pack:  
https://github.com/UtochkaI/johnny-telegram-bot-article

## Коротко

Я арендовал VPS под VPN, но в моём сценарии он оказался слишком проблемным: падения, DDoS-истории, техработы, слабая поддержка и периодически недоступный сайт провайдера. Деньги на балансе вывести было нельзя, поэтому я решил не бросать оплаченный сервер, а использовать его для чего-нибудь небанального.

Так начался эксперимент с Telegram-ботом. Сначала это была просто игрушка. Потом появился персонаж — Джонни, злой и уставший production-инженер, который ненавидит халтуру. А затем выяснилось, что главный вопрос в LLM-боте для группового чата — не “как заставить его отвечать”, а “как научить его вовремя молчать”.

В итоге проект вырос в набор инженерных механизмов: routing, suppress, cooldown, bot-share, reaction policy, image/vision route, dry-run, fake-client проверки, backup-дисциплина и regression matrix.

## Завязка: VPS, который не подошёл для VPN

Проект начался не с идеи сделать продукт и не с желания написать очередного AI-ассистента.

Я арендовал VPS у 62Yun под VPN. Для моей задачи он быстро перестал быть нормальной площадкой: падения, постоянные DDoS-истории, технические работы, поддержка, которая не особо помогает, и сайт провайдера, который сам периодически недоступен. Сервер был арендован на месяц, на балансе оставалось около 1200 рублей, вывести их было нельзя.

Бросать оплаченный ресурс было жалко. Поэтому я задал ChatGPT вопрос не в духе “как поднять ещё один сервис”, а примерно такой:

> придумай что-нибудь не банальное и забавное

Сначала это был просто эксперимент. Не архитектура продукта, не “личный AI-ассистент”, не open-source проект. Просто мысль: если сервер всё равно есть, пусть делает что-нибудь смешное.

Так постепенно появился Telegram-бот Джонни.

## Почему не ещё один вежливый помощник

Главная идея Джонни была не в функциях, а в характере.

Я не хотел делать бота, который на всё отвечает “конечно, вот несколько вариантов”. Таких ассистентов уже слишком много. Хотелось персонажа с инженерным темпераментом: резкого, прямого, злого на халтуру, но при этом технически полезного.

Джонни — это не саппорт и не корпоративный помощник. Его образ ближе к уставшему production-инженеру, который десять лет тушил чужие инциденты, видел все виды кривых деплоев, знает, чем заканчивается “давай быстро поправим прямо на проде”, и не хочет объяснять это в сотый раз.

На старте кажется, что такой характер — это просто prompt. Написал в system prompt нужный стиль, добавил пару фраз, получил персонажа.

На практике prompt оказался только верхним слоем. Он задаёт голос, но не управляет поведением в живом групповом чате. Для поведения нужны другие механизмы: routing, suppress, контекстные границы, память, cooldown, side-effect guards и регрессионные проверки.

## Первая рабочая схема

Базовая архитектура была простой:

```text
Telegram update
    -> bot runtime
    -> routing decision
    -> prompt/context assembly
    -> LLM API
    -> postprocess
    -> Telegram send / reaction / file / image
```

На этом этапе легко обмануть себя. Бот отвечает, стиль узнаваемый, сообщения смешные — значит, всё работает.

Но Telegram-группа быстро показывает, что “бот умеет отвечать” — это только малая часть задачи. В реальном чате появляются другие вопросы:

```text
когда отвечать?
когда молчать?
что считать обращением к боту?
что считать продолжением разговора?
что делать с коротким “пон”?
как не отвечать на старый контекст?
как не начать спорить с другим ботом?
как не захватить чат?
```

С этого момента проект перестал быть игрушкой.

## Cooldown: задержать — не значит потерять

Один из первых практических конфликтов был связан с cooldown.

Если бот отвечает слишком быстро и слишком часто, он выглядит неестественно и начинает доминировать в чате. Логичное решение — задержки и cooldown. Но простой cooldown легко ломает поведение.

Пользователь обращается к боту. Сообщение попадает в период охлаждения. Наивная реализация просто режет задачу. В результате бот либо молчит на прямой вопрос, либо позже использует этот контекст в другом ответе. Для пользователя это выглядит как игнор или шизофрения.

Поэтому появилась логика pending replies:

```text
плохо:
addressed message + cooldown -> drop

лучше:
addressed message + cooldown -> pending reply -> delayed send
```

Это был первый важный урок проекта: ограничитель активности не должен уничтожать адресную задачу. Он должен управлять темпом, а не ломать семантику диалога.

## TARGET_MESSAGE против RECENT_CONTEXT

Следующая боль — контекст.

LLM хочется дать историю чата, чтобы ответ был живым. Но если сделать это грубо, модель начинает отвечать не на текущее сообщение пользователя, а на что-то из недавнего фона.

В групповом чате фон почти всегда шумный: параллельные реплики, чужие вопросы, ответы других людей, мемы, технические обсуждения. Бот должен учитывать контекст, но не должен превращать его в задачу.

Так появилось жёсткое разделение:

```text
TARGET_MESSAGE = сообщение, на которое нужно ответить
RECENT_CONTEXT = фон, который помогает понять ситуацию
```

Звучит очевидно, но многие странности LLM-ботов в чатах начинаются именно с нарушения этой границы. Модель видит контекст и начинает “помогать” не тому человеку и не по тому вопросу.

В Джонни это стало отдельным направлением: target/context hardening, contextual reference resolver, semantic continuity, защита от stale context и привязка памяти к участнику.

## Самая сложная функция — молчание

Когда бот становится достаточно живым, пользователи сначала радуются. Потом админы начинают писать, что его стало слишком много.

Это был важный поворот. До этого фокус был на том, чтобы бот хорошо отвечал. После этого фокус сместился: бот должен уметь не отвечать.

Появились разные уровни ограничений:

```text
routing      — понять, адресовано ли сообщение боту
suppress     — решить, что отвечать не надо
cooldown     — ограничить темп
bot-share    — ограничить долю сообщений бота в чате
anti-chain   — не превращать user <-> bot в бесконечную цепочку
low-value    — не отвечать текстом на “пон”, “спс”, “ясно”
reaction     — иногда поставить реакцию вместо сообщения
```

Снаружи это выглядит как UX-настройка. Внутри это уже policy engine.

Особенно неприятный класс багов — когда suppress режет нормальный follow-up. Пользователь задаёт короткий, но валидный вопрос после ответа бота, а система считает это шумом и молчит. Для пользователя это выглядит не как “умный suppress”, а как тупой игнор.

Поэтому routing и suppress пришлось разделять по слоям. Нельзя лечить плохую маршрутизацию стилем ответа. Нельзя компенсировать плохой suppress prompt-инструкцией. Если бот не должен был молчать, это ошибка routing/suppress, а не “модель не так сформулировала”.

## Реакции: маленький side effect, много последствий

Отдельная ветка — реакции.

Казалось бы, что может быть проще: пользователь пишет “пон, спс”, бот ставит реакцию и не засоряет чат.

На практике реакции оказались полноценным side effect. Это публичное действие в Telegram. Его нельзя делать по слабому словарному триггеру, нельзя ставить “ок” на всё подряд, нельзя давать реакцию там, где нужна текстовая помощь.

Одна из ошибок процесса была в том, что реакционное поведение сначала пытались чинить через лексические признаки. Это быстро превращается в хрупкую конструкцию: слишком много “если слово такое — сделать это”. Потом стало очевидно, что реакция должна зависеть не от слова, а от ситуации и уверенности.

Итоговая логика стала такой:

```text
low-value acknowledgement + high confidence -> reaction
unclear short message -> clarify or suppress
actual task/question -> generate answer
```

Ещё один урок: даже маленькая реакция — это не “мелочь интерфейса”, а публичный side effect. Значит, ей нужен guard, dry-run и регрессионная проверка.

## Image/Vision: когда бот начал делать больше, чем писать

Потом появилась генерация изображений. Это резко подняло уровень риска.

Текстовый ответ — один side effect. Генерация изображения — другой класс: внешний API, файл, отправка в Telegram, задержки, стоимость, cooldown, включение/выключение фичи, защита от случайного запуска.

Для image route появились отдельные правила:

```text
image request должен быть явно адресован
naked image request не должен случайно проходить
генерация должна иметь enabled-gate
image API должен иметь dry-run
Telegram file send должен иметь fake-client проверку
служебные маркеры не должны попадать в историю
```

Потом добавился vision-контекст: бот начал учитывать сгенерированные и входящие изображения. И тут снова возникла старая проблема, только в медиа-варианте: stale context. Если пользователь прислал новое изображение, бот не должен отвечать по старой картинке. Если reply идёт к изображению, контекст должен быть привязан именно к нему.

Так текстовый бот постепенно стал системой с несколькими типами памяти и несколькими типами side effects.

## Production boundary: до live, а не после поломки

Один из важных выводов проекта: любую новую подсистему с side effect нужно пропускать через production boundary checklist до live-теста.

Для Джонни это стало особенно заметно на Telegram send, file send, reactions, image generation и semantic live behavior. Ошибка в этих местах видна пользователям сразу.

Минимальный набор проверок стал выглядеть так:

```text
- py_compile
- CLI smoke
- fake-client / dry-run
- no public send in dry-run
- no marker leak into history
- no DB/history pollution
- fresh bot.log check
- controlled live-smoke only after dry-run
```

Это скучная часть, но именно она отличает “скрипт на вечер” от сервиса, который можно держать в живом чате.

## Почему py_compile не спасает

`py_compile` полезен, но он ловит только синтаксис. Он не доказывает, что конкретная runtime-ветка работает.

В проекте был показательный случай: фикс реакции прошёл компиляцию, но в live path упал из-за `NameError`, потому что в конкретном scope не было нужной переменной. Файл был синтаксически валиден, но execution path не был проверен.

После такого становится очевидно, что закрывать патч только по `py_compile` нельзя. Нужна матрица поведения.

Пример regression matrix для такого бота:

```text
addressed high-priority question -> answer
reply follow-up -> answer or delayed answer
low-value reply-to-bot -> reaction/suppress by policy
naked trigger -> no spam
image request -> image route only when addressed
semantic live -> no duplicate reply
dry-run -> no Telegram send / no image API / no DB leak
bot.log -> no fresh ERROR/Traceback
```

Главный неприятный вывод: если такой матрицы нет, владельцу проекта приходится становиться live-QA. Он сам в чате ловит регрессии, которые должны были быть пойманы до рестарта сервиса.

## Layer discipline

Со временем появилась дисциплина слоёв. Перед патчем нужно определить, где проблема:

```text
routing / suppress / memory / context / prompt / reference / style /
side-effect / image / reaction / DB / CLI / service
```

Это не формальность. Если бот отвечает не тому сообщению, проблема не в стиле. Если suppress режет нормальную задачу, это не лечится prompt-ом. Если реакция улетает публично не туда, это не “тон ответа”, а side-effect boundary.

Такая дисциплина защищает от типичного хаоса LLM-проектов, где всё пытаются решить новой инструкцией в prompt. Prompt важен, но он не должен быть помойкой для компенсации плохой архитектуры.

## Что получилось в итоге

Снаружи Джонни — Telegram-бот с резким характером. Внутри — набор инженерных механизмов:

```text
- routing по обращению, reply, mention, trigger
- suppress для low-value и шумовых сообщений
- cooldown и bot-share ограничения
- participant-bound memory
- TARGET_MESSAGE / RECENT_CONTEXT separation
- contextual reference resolver
- reactions как отдельный side effect
- image generation и vision context
- dry-run/fake-client проверки
- backup и service discipline
- regression matrix после рискованных патчей
```

Самое интересное оказалось не в том, что LLM умеет смешно отвечать. Это ожидаемо. Интересное — в том, сколько инженерии появляется вокруг одного “живого” бота, если пустить его в реальный групповой чат.

## Вывод

LLM-бот в Telegram — это не просто prompt engineering.

Prompt задаёт характер. Но устойчивое поведение задают другие вещи: маршрутизация, подавление лишних ответов, границы контекста, контроль side effects, наблюдаемость, тесты, backup и дисциплина изменений.

Плохой VPS для VPN в итоге оказался полезен: он стал полигоном для эксперимента, который быстро вырос из “сделай что-нибудь забавное” в полноценный разбор того, как LLM ведёт себя в живой социальной среде.

Главный урок получился простой: если бот умеет только отвечать — он ещё не готов к группе. Сначала его нужно научить молчать.
